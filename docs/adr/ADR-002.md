# ADR-002: Sigurnosne mjere web aplikacije

## Status

Prihvaćeno

## Datum

2026-05-30

## Kontekst

NoteApp prima korisnički unos kroz HTML obrasce i JSON API zahtjeve. Podaci se spremaju u MySQL bazu i prikazuju drugim dijelovima aplikacije, pa su najvažniji rizici SQL injection, cross-site scripting, neovlašteno slanje HTML obrazaca, neispravni ulazni podaci i neprimjerena pohrana lozinki.

## Odluka

Implementirane su sljedeće sigurnosne mjere:

- svi SQL upiti koriste PDO prepared statements i parametre,
- korisnički podaci u HTML predlošcima ispisuju se kroz `htmlspecialchars`,
- HTML obrasci koriste CSRF token iz sesije i serversku validaciju tokena,
- bilješke se validiraju na serveru: naslov je obavezan i ograničen na 255 znakova, a sadržaj ne smije biti prazan,
- HTTP odgovori sadrže Content-Security-Policy zaglavlje,
- lozinke se pri registraciji spremaju isključivo kao hash dobiven funkcijom `password_hash`, a model odbija spremanje vrijednosti koja nije hash.

## Razlozi

Prepared statements uklanjaju potrebu za ručnim slaganjem SQL upita i sprječavaju SQL injection. Escaping u HTML-u smanjuje rizik od XSS napada kada se prikazuju naslovi, sadržaji, kategorije ili imena korisnika. CSRF tokeni štite HTML obrasce od neželjenih zahtjeva pokrenutih s druge stranice. Serverska validacija osigurava da pravila vrijede neovisno o klijentskom sučelju. Content-Security-Policy dodatno ograničava izvore skripti, stilova i drugih resursa. Hashiranje lozinki štiti korisnike ako dođe do neovlaštenog pristupa bazi.

## Alternative

Razmatrane alternative:

- Ručno escapati vrijednosti prije SQL upita: odbačeno jer je sklonije pogreškama od prepared statements-a.
- Osloniti se samo na HTML `required` i `maxlength`: odbačeno jer se klijentska validacija može zaobići.
- Ne dodavati CSRF zaštitu na forme koje šalju Fetch zahtjeve: odbačeno jer su to i dalje HTML obrasci koje aplikacija prikazuje korisniku.
- Koristiti vanjsku sigurnosnu biblioteku: nije potrebno za trenutni opseg jer PHP i PDO već daju potrebne osnovne mehanizme.

## Posljedice

Pozitivne posljedice:

- SQL upiti više ne ovise o ručno interpoliranim vrijednostima,
- prikaz korisničkog sadržaja sigurniji je u HTML stranicama,
- neispravni obrasci vraćaju jasne poruke bez spremanja podataka,
- lozinke ostaju pohranjene kao hash.

Negativne posljedice:

- API zahtjevi za prijavu i registraciju iz web sučelja moraju poslati CSRF token,
- CSP trenutno dopušta inline stilove zbog postojeće upotrebe CSS varijable za boju kategorije.
